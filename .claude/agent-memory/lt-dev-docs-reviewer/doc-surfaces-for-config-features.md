When a branch adds a new configurable feature or core module, these doc surfaces all need updating (per CLAUDE.md "Living Documentation" + .claude/rules/). Check each:

1. `src/core/modules/<mod>/README.md` — module overview + usage
2. `src/core/modules/<mod>/INTEGRATION-CHECKLIST.md` — required for every core module needing project integration (.claude/rules/core-modules.md)
3. JSDoc on the new config interface(s) in `src/core/common/interfaces/server-options.interface.ts`
4. `.claude/rules/configurable-features.md` — TWO tables: the per-feature "Applied Features" row AND (if it adds overrides) the "Module Override Pattern" / ICoreModuleOverrides table. Keep both consistent with each other AND with the source interface.
5. `FRAMEWORK-API.md` — regenerated by `pnpm run build`, but only expands interfaces in the generator allowlist (see [[framework-api-generator-allowlist]]).
6. `.env.example` — new env vars (`AI_*`, `NSC__*`, etc.). Easy to miss; README/CHECKLIST mentioning them is NOT a substitute — deployers scan .env.example.
7. `docs/REQUEST-LIFECYCLE.md` — only if new controllers/resolvers/interceptors/middleware/routes change the request flow.
8. `migration-guides/` — required when a developer must do more than `pnpm update` to use/accommodate the feature (a new opt-in module with new env vars + a main.ts edit for MCP OAuth qualifies). See .claude/rules/migration-guides.md.

**Common consistency trap:** the override-field list often diverges across (a) the runtime `CoreXxxModule.forRoot()` options interface, (b) the public `ICoreModuleOverrides.<mod>` type, and (c) the two configurable-features.md tables. core.module.ts typically spreads `...overrides?.<mod>` into forRoot, so runtime may accept MORE fields than the public type declares — meaning some documented/runtime fields are not passable via the typed API. Cross-check all three.
